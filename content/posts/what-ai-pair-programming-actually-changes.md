---
title: What AI Pair Programming Actually Changes
tagline: Spoiler — it's not the code
date: 2025-04-02
---

I've spent several months using AI coding tools heavily — across Python, Java, and infrastructure work. Here's what I've actually noticed.

### The bottleneck moves

Before: I spent a lot of time writing boilerplate, looking up syntax, and translating intent into code.

After: I spend almost no time on those things. The bottleneck is now upstream — in the quality of my thinking before I touch the keyboard.

This is not a small shift. It means that knowing *what* to build, and *why*, matters more than ever. The execution gap between a senior and junior engineer used to partially mask differences in design thinking. That gap is closing fast.

### It surfaces bad architecture immediately

When you can generate an implementation in seconds, you discover very quickly whether your design is sound. If you find yourself fighting the AI, constantly correcting and re-prompting, it's usually a signal that the problem is under-specified or the design has a flaw.

Good abstractions feel effortless to implement with AI assistance. Bad ones feel like you're pushing uphill.

### What it hasn't changed

Code review still requires careful human judgement. The AI will confidently produce plausible-looking code that is subtly wrong in ways that only domain knowledge can catch.

And it still can't tell you what the right thing to build is. That remains stubbornly human.

### The management implication

If you're leading engineering teams, the question is no longer "can your engineers write code?" It's "can your engineers think clearly about problems?" The leverage on good thinking has just gone up by an order of magnitude.

That changes hiring, onboarding, and how you run technical conversations. Worth taking seriously.
